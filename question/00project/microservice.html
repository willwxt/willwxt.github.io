      # 微服务问题

      • 整个项目

      负载均衡

      网关 - 鉴权, 限流(guava limit), 黑白名单, 路由转发,

      开放平台: redis(userId+url+次数, 过期时间, 限制次数), (jwt, Oauth2.0)

      业务层: 企业支撑服务

      能力层/基础服务 (中台服务): 指令服务, 验签服务, 文件服务...

      注册中心: 调用者总是从注册中心获取服务列表

      配置中心: 服务配置的管理, 支持动态更新, 灰度发布, 切换环境

      中间件: redis, mq

      环境: linux, 虚拟机, docker

      一些问题

      • 数据库脱敏: 对称加密, 秘钥存注册中心
      • 三级等保: e.printStack()不打印堆栈信息, 不能使用sout, 不能打印秘钥信息
      • 服务优雅停机: 先把服务停机(降级, ), 再发下线请求从注册中心下线.

      # 压测问题

      1. 预估用户量, 通过28原则(80%的人在20%的时间访问), 计算QPS = 用户量/时间
      2. 压测环境: 必须压测试环境, 仿真环境, 生产环境
      3. 生产环境压测数据怎么办?
        1. 沙箱
        2. is_qa设置为1, 是否是qa的数据, 可删可不删
        3. 大公司用流量镜像

      # 三级等保

      1. 物理安全(除了阿里云, 还需要拥有自己的机房)
      2. 网络安全
      3. 主机安全(中高漏洞扫描)
      4. 应用安全(sql注入, 代码安全, 跨站脚本, 网页篡改, 敏感信息, 简单密码)
      5. 数据安全 (多地备份)
      6. 移动端安全 (加壳, 代码混淆, 交互通过HTTPS)
      7. IOS需要怎么上线(如何过审? 连接仿真环境造假数据, 过审后再切回生产环境)

      有什么坏处吗? 系统复杂性提高, 可靠性降低, 一致性问题
