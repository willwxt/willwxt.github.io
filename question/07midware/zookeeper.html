# zookeeper

概述: 分布式协调服务, 提供的一些简单的操作(原语集), 但是通过组合这些操作可以实现复杂的功能

基于zk实现诸如数据发布/订阅、负载均衡、命名服务、分布式协调/通知、集群管理、Master 选举、分布式锁和分布式队列等功能。

相关CAP理论, 保证了CP: 分区容错性和一致性

几个特点 :

  1. zk是主从的, 拥有leader和follower两种角色, 每个实例都是全量拷贝
  2. leader负责写, follower负责读, 客户端可以访问所有follower, 只不过读请求会被路由到leader执行
  3. 既然是主从, 肯定会有主机的单点故障, 导致集群不可用; zk的单点故障快速恢复机制保证集群在200ms内重新选举恢复可用状态

# zookeeper数据结构/数据模型/分层命名空间

  • 分成的命名空间: znodes
  • zookeeper 提供的名称空间非常类似于标准文件系统, key-value 的形式存储
  • key 是由斜线 / 分割的一系列路径元素

# 保证

  • 顺序一致性 - 来自客户端的更新将按发送顺序应用(单机leader, Zxid实现)
  • 原子性 - 更新要么成功要么失败
  • 单一系统映像 - 无论连接到哪个服务器, 客户端都将看到相同的服务视图
  • 可靠性 - 应用更新后, 它将从那时起一直存在, 直到客户端覆盖更新, 支持持久化
  • 及时性 - 对follower可以使用sync从leader同步数据, 获得最新数据, 保证数据一致性

# 简单使用

  • 只有启动3台, 才能过半投票选举成功
  • 启动时都需要从leader拉取快照
  • Zxid: zk事务id : 例如0x200000002 : 前面0x2是纪元, 后面是事务序列
    • cZxid: create创建事务id
    • mZxid: modify修改事务id
    • pZxid: 该节点及其子节点最新创建的事务id
    • 事务id的消耗: session建立销毁, zk节点up都会占用Zxid
  • 会话级别的临时节点, 并且可以共享视图, 可以应用实现同步功能/锁功能
  • sessionid以及视图: sessionid可以共享+漂移
  • sequential统一命名 create -s
  • 通过以上概念可以组合实现锁, 队列事务锁, HA, leader选举

# 集群知识

  • 3个才能开启集群
  • 2888: leader开启服务, 让follower来连接, 并接受follower的write事件
  • 3888: follower开启服务, 其他所有节点都回连接上, 主要用于选主投票

# Observer

其实一共有三个角色: Observer: 没有选举权限, 只能查询

比如30台机器可以奇数如15台follower, 其他全是Observer
