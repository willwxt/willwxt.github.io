      # 为什么使用消息队列? 消息队列的优点和缺点? kafka、activemq、rabbitmq、rocketmq都有什么优缺点?

      你们系统里为什么要用消息队列这个东西, 有什么好处; 既然用了, 那当时为什么选用这一款MQ

      1. 为什么使用消息队列(项目中什么业务场景, 使用MQ带来了很多好处):
      2. 消息队列主要有三个功能, 解耦 / 异步 / 削峰
        1. 解耦: 当我们系统需要调用多个系统, 调用复杂, 此时不需要实时返回的, 可以用MQ异步解耦
        2. 异步: 当我们调用比较耗时的模块, 使用消息中间件慢慢消费, 异步返回结果
        3. 削峰: 高峰期, 将请求都压入消息中间件(消息堆积), 避免后台数据库压力过大
      3. 缺点 (优点上面就是了)
        1. 系统可用性降低: 引入的依赖越多, 系统可用性越低; 如何保证消息一致性? MQ系统可用性?
        2. 系统复杂性提高: 消息丢失? 消息重复消费? 如何保证顺序性?
        3. 数据一致性
      4. 引入确实带来很多优势和麻烦, 但是该用的时候还是得用

      # kafka、activemq、rabbitmq、rocketmq都有什么优缺点？

      1. 吞吐量: kafka > rocketmq > rabbitmq > activemq
      2. 时效性: RabbitMQ(微秒级) > 其他(毫秒级)
      3. 可用性: 主从镜像(activemq, RabbitMQ) 分布式架构(rocketmq, kafka)
      4. 消息丢失: 可以做到不丢失(rocketmq, kafka, RabbitMQ)
      5. 协议: 支持多协议(RabbitMQ) , 自定义协议(kafka)
      6. activemq: 功能成熟, 高并发用的少
      7. RabbitMQ: 支持很多协议, 比如物联网, erlang开发较难定制/二次开发/不确定性
      8. rocketmq: 社区阉割版, 有精力二次开发的可以选择
      9. kafka: 仅提供核心功能, 很多功能需要自己实现, 提供超高吞吐量

      # 如何保证消息中间件的高可用性?

      集群

      1. RabbitMQ普通集群: 就是启动多个不同的RabbitMQ, 主要提高吞吐量,
      2. RabbitMQ镜像集群: 一主多副本集群, 副本只做镜像复制, 读写都在主机
        1. 新增镜像模式策略, 设置数据同步到指定数量的节点, 后续创建queue会自动同步
        2. 主机处理消息需要同步所有副本
      3. kafka高可用
        1. 每个topic可以指定多个partition, 每个partition可以指定多个副本(一般3个足够)
        2. 相同partition的副本一般均匀分布在不同broker上, leader同理
        3. 生产者推送消息到leader, 根据策略同步到其他副本, 消费者最多消费到HW

      # 如何保证消息不被重复消费? (幂等性)

      本质是业务架构如何保证在使用mq时候的幂等性, 主要由业务保证的

      kafka本身的一个幂等是保证生产者推送消息重试时(单分区中)不会发生消息重复

      1. 首先为什么会重复消费: 一般是先处理业务, 然后再向消息中间件 ack / 保存消费位移, 但是反过来则可能消息丢失
      2. 主要根据实际业务来定制解决方案: 比如在消息流转中增加全局唯一id, 根据id来判断是否消费过

      # 如何保证消息的可靠传输? (消息丢失)

      1. RabbitMQ:
        1. 生产者丢失: 提供事务机制(同步, 开销大), 提供confirm确认机制(根据投递结果返回ack/nack, 允许异步)
        2. RabbitMQ自己弄丢数据(未持久化): 跟前面ack配合, 持久化到磁盘后再回复生产者ack
        3. 消费端丢失: 消费端自动ack问题, 可以开启手动ack, 这样就可以先处理业务再ack回复
      2. kafka:
        1. 生产者: 设置参数 acks = -1, 那么收到消息的leader必须同步副本, 可以保证不丢失
        2. kafka丢失数据: 先保证kafka集群副本足够可靠好吧, 足够ISR作为可用副本
        3. 消费端: 同样, 关闭自动提交消费位移; 先消费, 再提交消费位移

      # 如何保证消息顺序性?

      1. 首先, 生产者自己先去保证推送消息是顺序的好吧;
      2. RabbitMQ中, 在单个queue队列中是保证顺序的, 用一个消费者去消费这个queue, 消费者再顺序分发消息到不同的worker进行消费
      3. kafka是保证单个partition中消息的顺序性,
