# 为什么使用inverted-index倒排索引?

# 优势

  • 传统线性查找, 如果关键字在文档最后(而且文档内容很大), 需要较长时间
  • 倒排索引: 记录每个词条出现在哪些文档, 以及文档中的位置, 那么根据查询信息快速定位到相关词条的文档和位置

# 创建inverted-index顺序, 基于lucene

  1. 创建文档列表
    • 首先对原始文档数据进行编号 (DocId) 形成文档列表
  2. 创建倒排索引列表
    • 对文档中的数据进行分词, 得到词条;
    • 对这些分词/单词进行编号, 创建索引; 然后每个单词对应记录包含这个单词的所有文档编号

# 查询过程

  1. 用户输入查询内容
  2. 将查询内容进行分词, 得到用户要搜索的词条
  3. 用词条去倒排索引中匹配, 可以找到包含这些词条的所有文档编号
  4. 根据文档编号去文档列表中找到文档

# ES集群基本结构/要素

  • 集群cluster (拥有状态, status 字段指示着当前集群在总体上是否工作正常
    • green 所有的主分片和副本分片都正常运行。
    • yellow 所有的主分片都正常运行，但不是所有的副本分片都正常运行。
    • red 有主分片没能正常运行。
  • 节点nodes
    • 集群中可以拥有一个或多个节点
    • 节点有主节点和其他节点之分, 主节点负责管理集群 (增加删除索引/节点)
    • 所有节点都可以查询回数据
  • 分片shards
    • 创建索引时会创建分片(可指定分片数量和分片的副本数量)
    • 分片有主分片和副本分片
    • 所有被索引的文档都将会保存在主分片上，然后并行复制到对应的副本分片上
    • 我们既可以从主分片又可以从副本分片上获得文档

# 动态调整副本分片数目

将原来的1个副本配置, 变更为2副本

  • 相同节点数目的集群上增加更多的副本分片并不能提高性能
  • 提高了数据冗余量, 我们可以在失去2个节点的情况下不丢失任何数据

# 应对故障

我们关闭 NODE1 之后

  1. 我们关闭了主节点, 所以必须选举一个新的主节点 (NODE2)
  2. 同时失去了主分片 1 和 2, 此时集群状态本来应该是 red
  3. 但是其他节点上还存在 1 和 2 的完整副本, 所以新的主节点会立即将存活节点上的对应副本提升为主分片
  4. 所以, 此时集群的状态应该为 yellow
