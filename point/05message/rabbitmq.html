# RabbitMQ

参考资料: 消息队列调研 (opens new window)

RabbitMQ是一个开源的AMQP实现，服务器端用Erlang语言编写，支持多种客户端。

AMQP (Advanced Message Queuing Protocol) 高级消息队列协议是应用层协议的一个开放标准，为面向消息的中间件设计。主要特征是面向消息、队列、路由（包括点对点和发布/订阅）、可靠性、安全。

# 重要概念

  • VHost 虚拟主机; RabbitMQ提供的环境, 保证多个vhost之间互相隔离
  • publisher 生产者
  • consumer 消费者; 一个客户端只会建立一条长连接connection, 通过信道监听queue队列
  • message 消息: 消息头 + 消息体 + 路由键(route_key) (也要指定交换机)
  • broker 消息服务器, 消息代理; 将消息发送给指定交换机
    • Exchange 交换机 连接数个队列,负责接收消息,按照绑定关系存入队列
    • Queue 队列 真正存储消息的模块
    • Binding 绑定 路由键与队列的绑定关系, 交换机和队列之间是多对多的绑定关系
  • connection 连接; 连接中有多个通道
  • channel 信道,通道; 实际传输数据的通道

# 交换机

交换器分4种：direct，headers，topic，fanout

  • direct 直连, 路由键绑定的完全匹配, 只发送到一个队列
  • topic 主题, 广播模式, 广播给绑定匹配的所有队列(绑定通配符 #: 0或多个单词; *: 一个单词)
  • fanout 扇出, 广播模式, 广播给所有队列
  • headers 直连, 不用路由键, 匹配header进行路由, 匹配条件更多, 但是性能差

# 消息确认ACK

消息丢失可能出现的位置:

  1. 生产者 >>> Broker
  2. Exchange >>> Queue
  3. Queue >>> Consumer

# 生产者消息确认

# 1.事务机制

生产者开启事务模式 > 发送消息 > 成功则提交, 失败则回滚

# 2.Confirm机制

开启消息id, 消息投递成功后, ack同时将id带回; 如果失败则返回nack; confirm可以异步回调

# 数据持久化

  1. 创建queue的时候设置为持久化, queue的元数据会被持久化
  2. 发送消息将deliveryMode设置为2, 消息持久化

# 消费者消息确认

消费者收到消息, 会发送ack给RabbitMQ, 然后RabbitMQ会把已ack的消息删除

  1. 默认情况会自动ack
  2. 可以关闭自动ack, 消费完毕后调用api手动ack
