    # 综述

    1. 首先zookeeper也是leader(写) - follower(读)模型, 那首先会有leader挂掉导致服务不可用的问题
    2. 但事实上, zk集群是极其高可用的, (如果有一种方式可以快速恢复leader)

    zookeeper 中两种运行状态: 可用和不可用 状态

    不可用状态恢复到可用状态应该越快越好

    • 官方压测: 选择新领导的时间不到200ms
    • 不同服务节点数, 根据读写比例的QPS 图
    • 数据模型和命名空间是一个目录树结构, 但是节点的概念, 节点可以存一些数据1M, 但是不要做数据库用!
    • session 会话, 每个请求来zk都有session
      • 持久节点
      • 临时节点 - session - 用来解决前面redis中说的死锁和锁时间的问题, session和锁可以同步生命周期
      • 序列节点/序列化概念 - 上两种节点支持序列化
    • 保证顺序(单leader好处就是容易保证写请求的顺序一致) 顺序一致性
    • 原子性 - 全部成功或全部失败
    • 单系统镜像 - 无论连接到哪个服务器, 都能得到相同的服务
    • 可靠性 - 持久化
    • 及时性 - 最终一致性

    # 节点属性

    cZxid 后32位创建操作的序列号, c = create, 前32位代表纪元, --- 本节点创建时候使用的全局序列号(事务ID)

    mZxid 修改操作

    pZxid 创建的最后节点的序列号

    ephemeralOwner 临时节点所有者(某连接的session)

    # 问题: 连接的节点挂机后, session是否会消失?

    session是所有客户端都有, 所以连接的节点挂机, 连接到别的节点依然会有原来的session

    client连接zk的时候, 创建的sessionid将会写给所有节点, 占用了一个全局序列号cZxid

    sessionid消失的时候, 也会占用一个全局序列号进行一次全局节点的删除操作

    # ZAB协议 - 在可用状态

    zookeeper 原子广播

    1. 客户端 create
    2. 客户端通知leader create
    3. leader 生成 Zxid
    4. leader 通知所有 follower 写日志log(向磁盘) (leader会维护一个发送队列, 保证有序)
    5. follower们会会送ok消息
    6. 当ok数量达到一半以上, 广播 write 指令, 将create数据写到内存
    7. follower收到write指令, 写完内存后, 返回ok
    8. 当leader收到过半的ok后, 给客户返回ok, 调用结束

    # zookeeper

    概述: 分布式协调服务, 提供的一些简单的操作(原语集), 但是通过组合这些操作可以实现复杂的功能

    基于zk实现诸如数据发布/订阅、负载均衡、命名服务、分布式协调/通知、集群管理、Master 选举、分布式锁和分布式队列等功能。

    相关CAP理论, 保证了CP: 分区容错性和一致性

    几个特点 :

    1. zk是主从的, 拥有leader和follower两种角色, 每个实例都是全量拷贝
    2. leader负责写, follower负责读, 客户端可以访问所有follower, 只不过读请求会被路由到leader执行
    3. 既然是主从, 肯定会有主机的单点故障, 导致集群不可用; zk的单点故障快速恢复机制保证集群在200ms内重新选举恢复可用状态

    # zookeeper数据结构/数据模型/分层命名空间

    • 分成的命名空间: znodes
    • zookeeper 提供的名称空间非常类似于标准文件系统, key-value 的形式存储
    • key 是由斜线 / 分割的一系列路径元素

    # 保证

    • 顺序一致性 - 来自客户端的更新将按发送顺序应用(单机leader, Zxid实现)
    • 原子性 - 更新要么成功要么失败
    • 单一系统映像 - 无论连接到哪个服务器, 客户端都将看到相同的服务视图
    • 可靠性 - 应用更新后, 它将从那时起一直存在, 直到客户端覆盖更新, 支持持久化
    • 及时性 - 对follower可以使用sync从leader同步数据, 获得最新数据, 保证数据一致性

    # 简单使用

    • 只有启动3台, 才能过半投票选举成功
    • 启动时都需要从leader拉取快照
    • Zxid: zk事务id : 例如0x200000002 : 前面0x2是纪元, 后面是事务序列
      • cZxid: create创建事务id
      • mZxid: modify修改事务id
      • pZxid: 该节点及其子节点最新创建的事务id
      • 事务id的消耗: session建立销毁, zk节点up都会占用Zxid
    • 会话级别的临时节点, 并且可以共享视图, 可以应用实现同步功能/锁功能
    • sessionid以及视图: sessionid可以共享+漂移
    • sequential统一命名 create -s
    • 通过以上概念可以组合实现锁, 队列事务锁, HA, leader选举

    # 集群知识

    • 3888: follower开启服务, 其他所有节点都回连接上, 主要用于选主投票

    # Zookeeper读/写流程

    # 读数据

    1. zookeeper 不保证读取的是最新数据 , 无论读取follower还是leader, 都是直接返回结果
    2. 如果要保证(follower)读取最新数据 , 需要使用 sync 方法 向leader同步数据

    # 写数据

    只要有 半数的节点同步成功  , leader就可以回复(ack)客户端写成功 , 剩下的节点只需要慢慢完成数据同步即可

    当 follower收到写请求 , follower将请求转发给leader , 完成后将回复ack给客户端

    # 服务器动态上下线

    1. 服务节点都向zookeeper上 注册临时节点 , 比如 /servers/server-0000001 , 顺便还能存点服务信息
    2. 客户端就可以从zk上, 根据存在的节点获取服务列表, 同时还可以监听这些节点
    3. 服务节点上线/下线后, 临时节点被删除, 监听器会通知客户端, 客户端重新获取服务列表

    # 做Master选举

    1. 服务器节点向zk上注册临时节点
    2. 这些节点尝试创建Master临时节点, 创建成功的作为Master, 其他为Slave
    3. 所有节点都监听Master临时节点的删除事件
    4. Master宕机则开启新一轮选举

    # zookeeper分布式锁

    1. 加锁: 在指定目录下创建一个 临时顺序节点, 多个取锁线程会逐渐创建一系列连续节点
    2. 判断自己创建的节点序号是不是 当前最小 , 如果是就获取到锁
      • 在这个规则下, 临时节点会按照序号逐个获取到锁
    3. 释放锁: 删除自己创建的节点

    # 面试重点

    # 选举机制

    1. 服务器id myid : 编号越大, 选举权重越大
    2. 事务id zxid : 编号越大, 数据越新
    3. 逻辑时钟 epoch-logicalclock : 同一轮投票值相同, 每一轮结束会递增
    4. LOOKING: 竞选状态
    5. FOLLOWING: 随从状态，同步 leader 状态，参与投票
    6. OBSERVING: 观察状态，同步 leader 状态，不参与投票
    7. LEADING: 领导者状态

    第一次, 服务器启动时选举流程

    1. 刚启动, follower节点进入looking状态
    2. 每个节点进行投票, 并把投票结果发送给集群中其他节点
    3. 收到投票结果的节点会进行比较
      1. 比较epoch
      2. 比较zxid
      3. 比较myid
      4. 刚启动时zxid和 epoch都一样, 所以 myid 大的胜出
    4. 统计投票信息 , 超过半数票的节点当选leader
    5. leader变为LEADING状态, follower变为FOLLOWING状态
