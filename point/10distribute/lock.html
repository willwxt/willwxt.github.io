分布式锁用于什么场景?

多机操作同一目标资源

# Redis分布式锁

锁过程要点如下

  1. 加锁: SET lock_key $unique_id EX $expire_time NX
    • 最初版: setnx 命令抢占锁 (SET if Not Exist) , 此时目标key (此处为lock_key) 作为锁
    • 解决死锁问题: 设置EX超时时间, 避免占锁线程挂掉, 导致的死锁
  2. 释放锁:
    • 解决释放别人的锁的问题: key中的value还没用到, 设置为UUID吧! 先 GET 判断锁是否归属自己, 再决定是否 DEL 释放锁, 整个过程为了保证原子性, 使用Lua脚本
  3. 锁续期 :
    • 解决锁过期时间不好评估的问题: 如果占锁线程正常, 定时对锁续期
  4. 仍然存在的问题: redis主从集群下, 数据不同步导致锁失效
    • 主机设置完锁后还未同步到从机就宕机了, 此时新主机根本没有上锁

# Redlock

至少要部署 5 个 Redis 实例, 而且都是主库, 它们之间没有任何关系

  1. 客户端先获取「当前时间戳T1」
  2. 客户端依次向这 5 个 Redis 实例发起加锁请求 (用前面讲到的 SET 命令)
    • 且每个请求会设置超时时间 (毫秒级, 要远小于锁的有效时间)
    • 如果某一个实例加锁失败 (包括网络超时、锁被其它人持有等各种异常情况) , 就立即向下一个 Redis 实例申请加锁
  3. 如果客户端在 >=3 个 Redis 实例加锁成功 (大多数加锁成功) , 则
    1. 再次获取「当前时间戳T2」
    2. 判断 T2 - T1 < 锁的过期时间, 如果还未超过过期(有效)时间, 则加锁成功, 否则加锁失败 (因为加锁过程需要一段时间, 如果直接超过了锁的有效时间, 则视为失败)
  4. 加锁成功, 去操作共享资源
  5. 加锁失败, 向「全部节点」发起释放锁请求 (前面讲到的 Lua 脚本释放锁)
# 特点
  1. 多个实例加锁: 容错, 部分机器宕机依旧可用
# 分布式问题

N P C: Net Delay网络延迟; Process Pause进程暂停(GC); Clock Drift时钟漂移

# 总结

  1. 所以，我对 Redlock 的个人看法是，尽量不用它，而且它的性能不如单机版 Redis，部署成本也高，我还是会优先考虑使用主从+ 哨兵的模式 实现分布式锁。
  2. 使用分布式锁，在上层完成「互斥」目的，虽然极端情况下锁会失效，但它可以最大程度把并发请求阻挡在最上层，减轻操作资源层的压力。
  3. 但对于要求数据绝对正确的业务，在资源层一定要做好「兜底」，设计思路可以借鉴 fecing token 的方案来做。

# fecing token 的方案，保证正确性

fecing token 方案保证分布式锁的正确性。

这个模型流程如下：

  1. 客户端在获取锁时，锁服务可以提供一个「递增」的 token
  2. 客户端拿着这个 token 去操作共享资源
  3. 共享资源可以根据 token 拒绝「后来者」的请求 (资源执行的token已经到了34, 那么33就不允许执行)

# zookeeper锁实现

分布式锁用Redis还是Zookeeper？ (opens new window)

  1. 客户端 1 和 2 都尝试创建「临时节点」，例如 /lock
  2. 假设客户端 1 先到达，则加锁成功，客户端 2 加锁失败
  3. 客户端 1 操作共享资源
  4. 客户端 1 删除 /lock 节点，释放锁

如果客户端 1 异常崩溃了，那么这个临时节点会自动删除，保证了锁一定会被释放。

客户端 1 此时会与 Zookeeper 服务器维护一个 Session，这个 Session 会依赖客户端「定时心跳」来维持连接。

# watch机制锁

zk基础知识, znode特性:

  1. 有序节点: 创建 /lock/node- 子节点时指明有序, 那么实际会顺序生成 /lock/node-0000000000
  2. 临时节点: 建立的临时节点, 在会话结束或者会话超时后, 会被 自动删除
  3. 事件监听: 对节点设置事件监听, 当节点数据或结构变化时, zk会通知客户端

zk分布式锁实现:

  1. 在 /lock/ 下创建临时+有序节点
  2. 获取 /lock/ 下所有节点, 判断当前创建的节点序号 是否最小
    • 如果最小, 认为获取成功
    • 如果不是最小, 则对前一个节点添加事件监听
  3. 释放锁就是将持有的临时节点删除, 于是就可以顺序触发下个节点的事件(锁获取成功)

# Zookeeper 和 redis 锁的优劣总结：

  1. 获取锁的方式
    1. redis获取失败, 需要重试获取, 而zk加锁失败只需要加上监听器进行watch, 不用一直轮询
    2. 但是监听器过多会导致zk集群压力较大
    3. zk不需要考虑过期时间
  2. 数据一致性
    1. redis不是强一致性, 主机宕机会导致锁失效
    2. zk强一致性, 允许宕机的数量更多, 可靠性更高
  3. 在架构本身有redis的情况下, 是否引入zk模块 (增加系统复杂性, 成本)
  4. 性能: 显然redis吞吐量更高
