      # 分布式事务

      分布式事务: 每个节点只知道自己的操作成功或失败, 无法知道其他节点的状态, 一个分布式事务跨越多个节点, 并且保证所有节点的事务ACID特性

      # XA规范和2PC和3PC

      XA规范: X/Open组织提出的规范, 使用两阶段提交协议(2PC)来保证分布式事务的原子性

      # 二阶段提交(Two-phase Commit, 2PC)

      引入一个作为协调者的组件, 来统一掌控所有节点(参与者)的操作结果, 并最终指示这些节点是否进行真正的提交(将数据写入磁盘);

      思路: 参与者将操作成败通知协调者, 再由协调者根据所有参与者的反馈情报决定各参与者要提交操作还是中止操作

      # 前提: 2PC成立基于以下假设

      1. 该分布式系统中, 存在一个节点作为协调者, 其他节点作为参与者. 且节点之间可以进行网络通信
      2. 所有节点都采用WAL, 即使节点损坏不会导致日志数据的消失
      3. 所有节点不会永久性损坏, 即使损坏后仍然可以恢复

      # 第一阶段(或称投票阶段, prepare)

      1. 协调者向所有参与者询问是否可以执行提交操作, 并等待节点响应
      2. 参与者执行询问发起的所有事务操作, 记录undo和redo
      3. 参与者响应询问: 如果事务执行成功, 响应同意; 事务失败则响应中止;

      # 第二阶段(commit)

      a. 成功: 当所有参与者都同意

      协调者发起提交 - 参与者提交- 参与者反馈完成 - 协调者收到所有提交完成后完成事务

      b. 失败: 任意参与者响应中止

      协调者发起回滚 - 参与者回滚 - 参与者反馈完成 - 协调者收到所有回滚完成后取消事务

      # 缺点

      1. 执行过程中, 节点都处于阻塞状态; 可能导致多个事务的连带阻塞
      2. 参与者崩溃, 需要协调者的超时机制进行回滚操作, 策略比较保守

      # 三阶段提交(Three-phase commit, 3PC)

      概述: 与二阶段提交不同的是, 三阶段提交是"非阻塞"协议

      1. 加入超时机制: 主要为了解决2PC的阻塞问题
      2. 将prepare阶段拆成 CanCommit和PreCommit:
        • CanCommit就是检测各事务是否可以成功获取锁(是否可以顺利执行事务)
        • 而PreCommit可以视为2PC的一阶段加上超时时间

      就是 Seata AT模式的 非数据库实现版本

      1. 每个模块都需要编写三部分代码: try(准备) confirm(确认提交) cancel(取消回滚)

      3. cancel中编写补偿逻辑

      # 可靠消息 + BASE

      处理服务先提交给消息服务, 确认发送后再通知消息服务进行真正发送

      # 最大努力通知型

      不保证数据调用成功, 但提供可查询的操作

      • 比如通过MQ进行通知(多次通知),查询校对(查证),对账文件,支付宝支付成功异步回调
