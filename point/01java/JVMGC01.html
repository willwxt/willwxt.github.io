# 垃圾

没有任何引用指向的对象

# java vs C++

GC处理垃圾

开发效率高, 执行效率低

手工处理垃圾

  • 忘记回收 - 内存泄漏
  • 回收多次 - 非法访问

开发效率低, 执行效率高

# 如何找到垃圾

# reference count 引用计数

# RC 不能解决的 : 循环引用

# Root Searching 根可达算法 背过

从 roots 开始找, 如过引用不到的视为垃圾

包括

  • JVM stack 线程栈(main线程栈帧里面引用的对象)
  • native method stack 调用C, C++本地方法中引用的对象
  • run-time constant pool 常量池, 类对象T.class
  • static references in method area 静态变量引用到的对象
  • Clazz

# GC Algorithms GC算法 背过

# Mark-Sweep 标记清理

标记可回收的, 然后回收

特点

  • 算法相对简单, 存活对象比较多的情况下效率较高
  • 两边扫描, 效率偏低, 容易产生碎片
# Coping 拷贝

将内存一分为二, 把搜索到的有用的对象全部copy到另侧, 直到最后清掉掉一侧

特点 适用于Eden区

  • 适用于存活对象较少的情况, 只需扫描一次, 没有碎片
  • 空间浪费(内存减半), 移动复制对象, 还要调整引用
# Mark-Compact 标记压缩

标记并在内存中从头排列

  • 不会产生碎片, 不会内存减半
  • 扫描两次, 移动对象, 修改引用

# 堆内存逻辑分区 (不适用不分代垃圾收集器)

# JVM分代模型

Epsilon, ZGC , Shenandoah 不使用逻辑分代模型

G1是逻辑分代, 物理不分代

# 逻辑分代模型

把内存分为 新生代 + 老年代 + 永久代(1.7)/元数据区(1.8)

新生代, 老年代属于堆

# 新生代 new/young
  • eden
  • survivor
  • survivor

默认比例 8:1:1

# 老年代 old/tenured

比例默认1:2 NewRatio java -XX:PrintFlagsFinal -version | grep NewRatio

# 永久代/元数据区 (method area 方法区)
  • 存放Class数据 代码编译完的信息 字节码 动态生成的class
  • 永久代必须指定大小, 而元数据允许不设置(无上限) - Perm Generation 容易溢出
  • 字符串常量放在永久代, 1.8开始放在堆

对象首先在栈上分配, 分配不下进入Eden区, 经过垃圾回收一次之后进入S1/S2(from/to) , 第二次垃圾回收后将Eden和S1的对象放入S2, 经过多次重复流程之后进入Old区

# GC概念 专业名词 YGC FullGC

Minor/YGC(Young GC) : 在年轻代进行回收的 , 新生代空间不足

MajorGC/FullGC : 整个区域进行回收, Old空间不足, System.gc()

# 栈上分配

线程私有, 小对象

无逃逸 - 在某一段代码块里, 作用域小

支持标量替换 - 简单对象, 可以用普通类型来代替这个对象

# 线程本地分配 Thread Local Allocation Buffer

防止多线程抢占Eden区内存位置, 给每个线程独有的内存空间(默认为Eden大小的1%) 避免线程间争用

# 常见的垃圾回收期 背过

# serial + serial Old

当GC干活的时候, 所有工作线程全部停止 STW (stop-the-world)

safe point 线程需要找到安全点才能停止 (马上停止可能会有问题的情况下)

serial 最早的垃圾回收器, 现在用的极少

# Parallel Scavnge + Parallel Old

如果上线没有对JVM做任何指定GC的设置, 默认的就是 PS + PO

也是STW , 不过是多线程的GC

# ParNew (Parallel New)

和 Parallel Scavenge 相似, 只是为了兼容CMS进行了优化, 可以和CMS某些阶段一起运行

ParNew VS Parallel Scavenge

ParNew 响应时间优先, 配合CMS

Parallel Scavenge 吞吐量优先

# CMS 非常重要

concurrent mark sweep 并发

并发回收, 并发标记和工作线程可以同时运行

并发回收是因为无法忍受STW

CMS 的四个阶段

  • 初始标记, 只标记根对象
  • 并发标记, 最浪费时间的, 标记现有能找到的对象(根可达)
  • 重新标记, 标记上个并发阶段发生改变的对象 (新的垃圾, 或者垃圾被重新引用, 取消垃圾标记)
  • 并发清理, 并发过程中产生的垃圾(浮动垃圾), 需要下次CMS清理

CMS 六个阶段还有预标记和整理

# CMS 缺点

# Memory Fragmentation 内存碎片

因为是Mark Sweep, 会产生内存碎片, 当内存存不下的时候会使用 Serial Old 进行压缩, 慢

# Floating Garbage 浮动垃圾

没有清理的浮动垃圾, 也一样会导致内存满, 要用 Serial Old 标记清理

  • Concurrent Mode Failure 解决方案, 降低触发CMS的阈值,

  • PromotionFailed

–XX:CMSInitiatingOccupancyFraction 92% 老年代占92%的时候产生FGC , 可以调低 保持老年代有足够的空间

# 并发标记的算法
三色扫描算法
	- 白 灰 黑
在并发标记时, 引用可能产生变化, 白色对象有可能被错误回收
解决方案
	SATB
	- snapshot at the beginning 起始时做一个快照
	- 当B->D消失时, 要把这个引用推到GC的堆栈, 保证D还能被GC扫描到
	Incremental Update
	- 当一个白色对象被一个黑色引用
	- 将黑色对象重新标记为灰色, 让collector重新扫描
	
  ---- (A黑)--  自己已经被标记, fields都标记完成
  |	  |		|
  V	  |		V
 (B灰) |	   (C灰) 自己标记完成, 还没来得及标记fields
  |	  |	
  V	  |	
 (D白)<- 没有遍历到的节点

# 常见垃圾回收期组合参数设定 (参考文档)

  • -XX:+UseSerialGC = Serial New (DefNew) + Serial Old
    • 小型程序。默认情况下不会是这种选项，HotSpot会根据计算及配置和JDK版本自动选择收集器
  • -XX:+UseParNewGC = ParNew + SerialOld
    • 这个组合已经很少用（在某些版本中已经废弃）
  • -XX:+UseConc(urrent)MarkSweepGC = ParNew + CMS + Serial Old
    • 红色部分有些版本需要加上
  • -XX:+UseParallelGC = Parallel Scavenge + Parallel Old (1.8默认) 【PS + SerialOld】
    • 这是jdk1.8默认的GC组合
  • -XX:+UseParallelOldGC = Parallel Scavenge + Parallel Old
    • 和上面其实是一样的
  • -XX:+UseG1GC = G1
  • Linux中没找到默认GC的查看方法，而windows中会打印UseParallelGC
    • java +XX:+PrintCommandLineFlags -version
    • 通过GC的日志来分辨
  • Linux下1.8版本默认的垃圾回收器到底是什么？
    • 1.8.0_181 默认（看不出来）Copy MarkCompact
    • 1.8.0_222 默认 PS + PO

# JVM调优

# 需要了解的基础概念

# 吞吐量

业务代码时间 / (业务代码执行时间 + 垃圾回收时间)

# 响应时间

STW越短, 响应时间越好

# 平衡以上两者关系

大数据计算, 数据挖掘, 注重吞吐量, 推荐 PS + PO

网站, api调用, GUI, 注重响应时间, 推荐 (1.8 G1) G1虽然响应时间快, 但是比较占用CPU资源, 吞吐量比CMS低15%

# 调优概念

  1. 根据需求进行JVM规划和预调优
  2. 优化运行JVM运行环境（慢，卡顿）
  3. 解决JVM运行过程中出现的各种问题(如OOM)
  4. 重启reboot

# 从规划开始

  • 调优从 业务场景, 业务流程 开始

  • 调优前提是能够从压力测试中看到结果...

  • 非要总结操作步骤的话:

    1. 根据业务场景, 选择合适的 垃圾回收器组合

      1. 业务偏向响应时间, 吞吐量, 还是两者兼顾

    2. 计算内存需求 (很难计算, 比较看经验值, 可能可大可小)

    3. CPU 越强越好

    4. 设定参数

      1. 年代大小, 升级年龄

      2. 日志参数

        1. -Xloggc:/opt/xxx/logs/xxx-xxx-gc-%t.log -XX:+UseGCLogFileRotation -XX:NumberOfGCLogFiles=5 -XX:GCLogFileSize=20M -XX:+PrintGCDetails -XX:+PrintGCDateStamps -XX:+PrintGCCause

          这里的命令是5个20M的日志文件循环使用(更新)

        2. 或者每天产生一个日志文件

    5. 观察日志情况

# 文档网站响应慢, 原 1.5G 扩到 16G 反而变卡

  1. 为什么原网站慢?

    很多用户浏览数据，很多数据load到内存，内存不足，频繁GC，STW长，响应时间变慢

  2. 为什么会更卡顿？

    内存越大，FGC时间越长

  3. PS + PO 换成 PN + CMS 或 G1

  4. 工作线程和 垃圾回收线程, 哪个占比较高

# card table

扫描根对象, 根对象可能指向了一个Old区对象, 然后Old区对象指回年轻代对象, ==>> YGC也要查Old区?

JVM将内存分成一个个card块, 如果Old区有card指向年轻代, 就标记为 dirty card , 下次扫描时只需要扫描Dirty Card , Card 的状态记录在 Card Table 中

# GC参数

  • -Xmn -Xms -Xmx -Xss 年轻代 最小堆 最大堆 栈空间 后跟200m

    • -XX:+UseTLAB 使用TLAB，默认打开
    • -XX:+PrintTLAB 打印TLAB的使用情况
    • -XX:TLABSize 设置TLAB大小
    • -XX:+DisableExplictGC System.gc()不管用 ，FGC

  • -XX:+PrintGC

  • -XX:+PrintGCDetails

  • -XX:+PrintHeapAtGC

  • -XX:+PrintGCTimeStamps

    打印各种GC信息

    • -XX:+PrintGCApplicationConcurrentTime (低) 打印应用程序时间
    • -XX:+PrintGCApplicationStoppedTime （低） 打印暂停时长
    • -XX:+PrintReferenceGC （重要性低） 记录回收了多少种不同引用类型的引用

  • -verbose:class 类加载详细过程

  • -XX:+PrintVMOptions

    打印JVM运行参数

  • -XX:+PrintFlagsFinal -XX:+PrintFlagsInitial 必须会用, 查找相关参数

  • -Xloggc:opt/log/gc.log

  • -XX:MaxTenuringThreshold 升代年龄，最大值15

    • 锁自旋次数 -XX:PreBlockSpin
    • 热点代码检测参数-XX:CompileThreshold
    • 逃逸分析 标量替换 ...

    这些不建议设置

# Parallel常用参数

  • -XX:SurvivorRatio

    Eden与S区比例, 默认 8:1:1

  • -XX:PreTenureSizeThreshold 大对象到底多大 , 过大直接分配到Old区

  • -XX:MaxTenuringThreshold

  • -XX:+ParallelGCThreads 并行收集器的线程数，同样适用于CMS，一般设为和CPU核数相同

  • -XX:+UseAdaptiveSizePolicy 自动选择/自适应 各区大小比例

# CMS常用参数

  • -XX:+UseConcMarkSweepGC 使用
  • -XX:ParallelCMSThreads CMS线程数量 默认使用一半 需要留给应用
  • -XX:CMSInitiatingOccupancyFraction 使用多少比例的老年代后开始CMS收集，默认是68%(近似值)，如果频繁发生SerialOld卡顿，应该调小，（频繁CMS回收）
  • -XX:+UseCMSCompactAtFullCollection 在FGC时进行压缩 作为解决碎片化的参数
  • -XX:CMSFullGCsBeforeCompaction 多少次FGC之后进行压缩
  • -XX:+CMSClassUnloadingEnabled 回收不用的class
  • -XX:CMSInitiatingPermOccupancyFraction 达到什么比例时进行Perm回收
  • GCTimeRatio 设置GC时间占用程序运行时间的百分比
  • -XX:MaxGCPauseMillis 停顿时间，是一个建议时间，GC会尝试用各种手段达到这个时间，比如减小年轻代

# G1常用参数

  • -XX:+UseG1GC
  • -XX:MaxGCPauseMillis 建议值，G1会尝试调整Young区的块数来达到这个值
  • -XX:GCPauseIntervalMillis ？GC的间隔时间
  • -XX:+G1HeapRegionSize 分区大小，建议逐渐增大该值，1 2 4 8 16 32。单位 m 随着size增加，垃圾的存活时间更长，GC间隔更长，但每次GC的时间也会更长 ZGC做了改进（动态区块大小）
  • G1NewSizePercent 新生代最小比例，默认为5%
  • G1MaxNewSizePercent 新生代最大比例，默认为60%
  • GCTimeRatio GC时间建议比例，G1会根据这个值调整堆空间
  • ConcGCThreads 线程数量
  • InitiatingHeapOccupancyPercent 启动G1的堆空间占用比例

# ZGC

# Colored Pointer + LoadBarrier

GC信息记录在指针上, 而不是记录在对象头部

指针更改后, 马上可以重用原来指向的内存空间

不支持32位系统, 不可用压缩指针, 取指针中前 19-22 共4位作为标记, 根据标记来决定是否做一些事情

这四位可以看做是不同空间的虚拟地址
