        # VFS 虚拟文件系统

        内存首先建立起VFS , 一棵目录树 , 树上的不同节点最终映射到不同的物理位置 , 每个不同的物理位置可以是具体的文件系统(fat32, ext3, ext4) , 或是网络上的节点, 或是虚拟的块设备

        文件系统的抽象, 封装不同的文件系统实现, 保证上层应用的统一的调用

        inode号 相当于每个文件的id(唯一id)

        pagecache 页缓存 4k

        1. 如果两个程序先打开同一个文件, pagecache可以做到共享
        2. dirty 如果程序读取后, 修改了某个数据, 对应的pagecache标记成dirty , 最终dirty会写入磁盘(写入的参数可以配置)
          1. 主动保存
          2. 被动保存, 让内核达到阈值后flush, 比如占用了百分之多少, 可能会发生数据丢失

        FD 文件描述符 seek

        • 最终是交给app使用的, 拥有指针seek, 可以读取pagecache中的不同位置

        # 指令块/代码块

        { echo "123"; echo "234"; }	# {  } 之间的空格不能省略, 可以执行多条指令 , ()也可以是代码块, 但是是子进程调用, {}是本进程

        # 系统调用 system call

        int 0x80 - 中断, cpu的指令, 一个值, - 去寄存器的中断向量表中查找 - 可以调用callback代码块

        用于切换用户态和内核态

        • app请求内核 - 触发中断 - cpu调用 - app保护现场 - kernel查找内存
        • 如果缺页 - 触发DMA - app挂起
          • dma完成 - 触发中断 - app运行
        • 否则直接返回

        # 第二节

        pagecache

        kernel 深入理解linux内核 深入理解计算机系统

        回顾: cpu mmu 的虚拟地址和物理地址的转换

        缺页: 因为程序是按需分配内存, 当程序运行过程中需要更多的内存, 产生缺页异常, 需要cpu再分配内存

        程序启动步骤: 读取程序文件 -- txt ,

        pcstat /bin/hash	# 程序消耗的页数, 缓存的页数
        1

        C程序会在内存中开辟 txt , data , heap , stack

        运行相同的程序, 共享使用pagecache, 可以只占用一份空间

        类似的, 相同文件被不同进程读到内存中也是使用相同位置的pagecache

        pagecache 内核维护的 中间层

        可以分配堆内堆外 (内外针对jvm而言, 但还是在java进程的堆中)

        拥有 pos limit capacity

        flip 设置limit, pos归零

        compact 压缩/整理数据, pos移到空位第一格, limit移到最后

        # RandomAccessFile , FileChannel

        seek

        getChannel 拿到 nio 的 FileChannel 文件通道 , 获得 MappedByteBuffer

        只有文件的通道才有 map (系统的mmap) , 得到一个和文件映射的内核pagecache (可以直接操作)

        原来必须要 write (系统调用/用户-内核态切换) , 现在直接操作pagecache (不需要系统调用, 但还是使用了pagecache, 有上述丢数据问题, java无法解决)

        • 可以用一些jni扩展库(c实现), 使用linux的Direct IO, 直接IO是忽略pagecache的 ; 需要自己写代码解决一致性和dirty等问题(自己管理pagecache) ; 可以自定义自己的pagecache...

        # TCP参数

        tcpdump -nn -i eth port 9090	# 监控, 抓取数据包
        1

        TCP 面向连接的可靠的传输协议 - 三次握手 - 双方在内核开辟资源 - 着两个资源代表了这个连接

        socket 是一个四元组 CIP CPORT SIP SPORT - 内核级 , 数据包可以通过这个四元组识别唯一的连接, 只要保持四元祖的唯一性, 可以在同个端口开启多个连接

        文件描述符是对socket的抽象, 应用是通过操作文件描述符进行通信

        # backlog

        核心允许再开两个TCP连接, 之后的不回复确认包, 进入 SYN_RECV, 连接失败

        # 窗口机制 缓冲区

        MTU : 网卡可以看到, 发送包的总大小

        MSS : 包体 (内容) 大小

        如果需要发送比较大量的数据 , 需要一次发送很多包, 但是每个包都要来回接受"我收到了"的通信, 会浪费较多时间, 所以实现了一个窗口机制, 可以连续发送一定数量的包, server按序处理

        但是如果server端的缓存(窗口?)满了, 之后发送的包就会丢失
