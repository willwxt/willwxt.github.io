                                              # DESIGN PARTTERNS

                                              # *单例模式

                                              8种方法, 安全的只有两种

                                              把构造方法设置为private

                                              恶汉式 懒汉式 CL DCL

                                              静态内部类在外部类被加载的时候是不会被加载的, JVM保证内部类只加载一次(静态内部类单例写法)

                                              枚举单例, 除了保证线程安全, 还可以防止反序列化

                                              一般交给Spring

                                              # 策略模式 Strategy

                                              Comparable

                                              要求对象继承该接口, 才能进行比较, 并不是策略模式. 因为如果想要使用不同的比较策略, 必须修改类, 违背了对更改关闭, 对扩展开放原则

                                              Comparator - 策略模式

                                              比较的同时需要实现比较器类, 通过实现不同的比较器, 可以定义需要的比较方式, 易于扩展

                                              简单的来说, 所有的 if判断/switch 的逻辑代码块都可以改为策略模式

                                              # 抽象工厂

                                              要求任意定义产品一族

                                              定义一个抽象工厂, 包含需要生产的产品一族

                                              所以当你需要生产一个系列的产品族, 只要从抽象工厂继承, 实现自己生产这些产品的逻辑即可

                                              最终调用的时候, 通过多种向上转型, 你只需要修改调用的工厂实现, 不需要修改任何代码

                                              # Proxy 代理

                                              静态代理, 动态代理, SpringAOP

                                              jdk的实现 决定了动态代理的类必须实现接口

                                              # 场景

                                              • 问题一: 计算坦克运行时间 (在不能改变源码的情况下)
                                              • 问题引申: benchmark , 记录调用模块的耗时
                                              1. 继承 (但是耦合度太高, 每个需求都要继承)
                                              2. 聚合, 代理类(维护一个目标类) , 不需要每次继承, 但是无法实现代理的各种组合
                                              3. 静态代理, 聚合类维护聚合类本身, 与目标共同继承与一个接口, 实现代理之间的组合使用

                                              很多设计模式只是语义上的区分, 代码非常类似(都是多态等), 比如装饰者可以视为代理

                                              • 问题二: 分离代理行为与被代理对象 (把打印日志的代理功能修改为可以代理所有类型/Object)
                                              • 动态代理 - 不是我们手写的代理类, 而是动态生成的代理类

                                              反射: 通过class字节码分析方法, 属性...

                                              Proxy.newProxyInstance(ClassLoader, Interfaces, InvocationHandler) 创建动态代理对象, 需要传入类加载器, 这个类实现的接口, 以及实现了代理方法的分装类

                                              可以配置参数, 将动态生成的类保存到磁盘 可以看到自动生成的动态代理类System.geProperties().put("jdk.proxy.ProxyGenerator.saveGeneratedFiles","true");

                                              # 分析动态代理生成的类

                                              首先该类继承自 Proxy , 构造方会法传入并维护 InvocationHandler (代理方法封装类), 当调用方法时会直接调用 invoke 方法 (你实现的代理方法)

                                              invoke(Proxy, Method, Object[]) 代理对象 调用的方法 方法参数

                                              调用生成的代理类的方法 - 代理类会调用实现的 InvocationHandler 中实现的 invoke 方法, 并把方法参数传入; 在invoke中, 会调用代理的逻辑, 以及原方法

                                              # Visitor 访问者

                                              # 概述

                                              • 在结构不变的情况下动态改变对于内部元素的动作
                                              • GOF典型案例: 编译器

                                              # 思路

                                              电脑由三个部件组成, 不同visitor访问需要执行不同的代码, 然后分别调用电脑的 accept() 接待方法

                                              电脑则是直接调用组件的 accept() 方法

                                              如果把判断逻辑写入到 各个组件的 accept() 方法中, 每当加入新的visitor类, 需要修改所有的accept方法

                                              所以我们把 accept 的逻辑全部放在 visitor 中实现, 所以所有组件的 accept 代码都是调用 visitor 的 visit 方法 (并把自身传入), 而 visit 方法由 visitor 本身来实现, 这样每有新的 visitor, 我们不需要修改任何组件的代码

                                              抽象出 visitor 接口, 接口中拥有 visitorCPU visitorMemory... 用来访问不同组件的方法

                                              # 问题

                                              以上描述的问题在于, 受访问对象的结构一般是固定的, 如果新增组件, 需要修改所有的 visitor

                                              # 编译器

                                              编译器构建出抽象语法数AST,

                                              • 按照传统模式, 对每个节点调用方法需要判断每个节点类型, 调用对应的方法做处理
                                              • 使用visitor模式, 树上所有节点只调用 accept 方法(接受一个visitor), 调用每个不同的visitor本身就具有的方法实现

                                              # Builder 构建器

                                              # 概述

                                              分离复杂对象的构建和表示

                                              同样的构建过程(接口中)可以创建不同的表示(简单的,或 复杂的接口实现)

                                              # 思路

                                              当一个对象非常复杂, (比如需要传入50个参数... ), 拆分内部的构建过程, 依次调用构建方法, 最后返回构建完毕的对象

                                              builder 模式经常使用链式编程的写法

                                              # Adapter 适配器

                                              # 概述

                                              Adapter / Wrapper(包装器)

                                              接口转换器

                                              # 举例

                                              国标接口转接头

                                              java.io : Reader/Writer 和 Steam 之间的转换 , 例如 InputStreamReader OutputStreamWriter

                                              jdbc-odbc : Java通过(JDBC-ODBC bridge)访问ODBC访问SQLServer

                                              ASM transformer - ClassReader 和 ClassWriter 之间的

                                              # 误区

                                              常见的Adapter类反而不是Adapter或者说也算转换器..., 如 WindowAdapter KeyAdapter

                                              WindowAdapter 是避免实现所有的接口方法实现的 空方法的实现类

                                              # Bridge 桥接

                                              • 双维度扩展

                                              # 概述

                                              分离抽象与具体

                                              用聚合的方式 (桥) 连接抽象与具体

                                              # 思路

                                              将礼物类抽取不同的特征, 并不断增加

                                              同时礼物的种类也在不断增加, 造成类的数量过多

                                              解决:

                                              在最初的抽象类中 维护一个具体的实现(当然这个具体的实现也是一个向上转型的接口), 当你想要创建对应类型的具体实现, 只需要将对应的实现传至对应类型的构造器

                                              • 在抽象和具体两个维度上同时进行多态, 并通过聚合的方式互相联系

                                              # Prototype 原型

                                              # 概述

                                              java自带原型模式 , 也就是 Object.clone()

                                              • 实现原型模式要实现标记型接口Cloneable, 一般会重新clone()方法, 但如果没实现接口, 调用时会抛出异常
                                              • 一般用于一个对象的属性已经确定, 需要产生很多相同对象的时候
                                              • 需要区分深/浅

                                              极少用到, 极少问到

                                              # 原clone方法

                                              将对象的内存直接拷贝一份

                                              是浅拷贝(浅克隆), 两份对象中的引用指向同一个对象

                                              # 深拷贝

                                              浅克隆的同时, 将内部存在的引用对象也调用克隆

                                              # String 需要克隆吗

                                              String类型不需要进行深克隆, 因为String在常量池, 本来就是共用的

                                              • 对克隆(不论深浅)前后的两个对象的String属性进行修改, 并不会相互影响
                                              • 但是使用 new String() , 在堆里面创建了指向String常量的对象, 则会被改变

                                              # TemplateMethod 模板方法

                                              • 钩子函数, 回调函数

                                              # 概述

                                              重写方法, 系统帮你自动调用的方法, 都可以称为模板方法

                                              在父类中有个方法A调用了很多其他的方法, 子类在继承的过程中不需要重写A或是所有的方法, 只需要按需重写自己需要的方法, 调用A时就会生效

                                              相当于定义了一个方法模板 A () , 其中调用了 op1 op2 ... , 只要继承并重写这些 op1 op2 .. 然后调用 A() 就可以

                                              # State 状态

                                              • 根据状态决定行为

                                              # 概述

                                              如果当一个类中的方法大都要通过判断一个状态来决定执行的代码逻辑, 我们把State状态抽取出来, 进行不同状态的State的方法实现, 而原来的类中只需要调用 State 的handler()方法即可

                                              GOF中案例 : TCP 连接的 State

                                              根据TCP连接的状态, 拥有不同的 Open Close 等方法的实现

                                              # 特点

                                              当Client中方法数量比较固定, 状态容易增加的时候使用

                                              如果需要增加方法, 那么每个State中都需要实现一遍, 而增加一种State则相当方便

                                              如果没有那么多状态, 或者状态固定, 也没有必要使用这种设计模式

                                              # 有限状态机 FSM

                                              • 高内聚, 低耦合

                                              # 开闭原则 OCP

                                              • Open-Closed Principle
                                              • 对扩展开放, 对修改关闭
                                                • 尽量不修改原来代码的情况下, 进行扩展
                                              • 抽象化, 多态是开闭原则的关键

                                              # 里式替换原则 LSP

                                              • Liscov Substitution Principle
                                              • 所有使用父类的地方, 必须能够透明的使用子类对象

                                              # 依赖倒置原则 DIP

                                              • Dependency Inversion Principle
                                              • 依赖抽象, 而不是依赖具体
                                              • 面向抽象编程 / 面向接口编程 (里面的实现可以随便换)

                                              # 接口隔离原则 ISP

                                              • Interface Segregation Principle
                                              • 每一个接口应该承担独立的角色, 不干不该自己干的事
                                                • Flyable Runnable 不该合二为一
                                                • 避免子类实现不需要实现的方法
                                                • 需要对客户提供接口的时候, 只需要暴露最小的接口

                                              # 迪米特法则 LoD

                                              • Low of Demeter
                                              • 尽量不要和陌生人说话
                                              • 在迪米特法则中, 对于一个对象, 非陌生人包括以下积累
                                                • 当前对象本身 this
                                                • 以参数形式传入到当前对象方法中的对象
                                                • 当前对象的成员对象
                                                • 如果当前对象的成员对象是一个集合, 集合中的也是
                                                • 当前对象所创建的对象
                                              • 和其他类的耦合度变低
